# Cursor за 20 секунд

**IDE** (как VS Code) + **ИИ**, который видит ваш проект.

- Открываете папку с кодом
- Пишете задачу в чат (`Cmd+I` / `Ctrl+I`)
- Агент читает файлы, правит код, запускает команды
- Вы смотрите diff и принимаете / отклоняете изменения

Сегодня разбираем **классический IDE** — редактор + сайдбар агента.

---

# Обычный чат vs ИИ-агент

| Обычный чат (ChatGPT и т.п.) | Агент в Cursor |
|---|---|
| Отвечает текстом | **Делает работу** в проекте |
| Не видит ваши файлы сам | Читает репозиторий, ищет по коду |
| Копируете код вручную | Правит файлы, показывает diff |
| Нет терминала | Может запускать команды, тесты |
| «Расскажи как сделать» | «Сделай и покажи результат» |

**Простыми словами:** чат — консультант. Агент — джун в вашем репозитории (под вашим контролем).

---

# Что агент умеет (и чего нет)

**Умеет**
- Найти, где в коде живёт фича/баг
- Предложить и внести правки
- Объяснить чужой код
- Набросать тесты, скрипты, README
- Помочь с git / PR (если попросите)

**Не умеет «магически»**
- Знать бизнес-контекст, который вы не написали
- Гарантировать 100% правильность — **всегда review**
- Заменить понимание системы вами

# Словарь вайбкодера (1/3): база

| Термин | Простыми словами |
|---|---|
| **IDE** | Программа, где пишут код (наш Cursor / VS Code) |
| **Репозиторий (repo)** | Папка проекта со всей историей изменений |
| **Git** | Система, которая помнит все версии кода |

| **Фреймворк / библиотека** | Готовый «конструктор», чтобы не писать с нуля |
| **Зависимости (dependencies)** | Чужие библиотеки, которые нужны проекту |

# Режимы чата — зачем их четыре

В одном и том же чате можно переключать **режим** — это стиль работы агента.

| Режим | Когда | Что делает |
|---|---|---|
| **Ask** | Вопрос, разбор, «как это работает?» | Читает и объясняет, **не меняет** код |
| **Agent** | «Сделай задачу» | Ищет, правит, запускает команды |
| **Plan** | Большая / неясная задача | Сначала план, потом реализация |
| **Debug** | Баг, ошибка, «не работает» | Ищет причину по симптомам и логам |

Правило: **неясность → Plan**, **вопрос → Ask**, **баг → Debug**, **ясно что делать → Agent**.

# IDE: куда смотреть на экране

Во время демо покажите:

1. **Файлы** слева — обычный проект
2. **Редактор** — код и diff от агента
3. **Чат агента** справа (`Cmd+I`) — сюда пишем задачи
4. **Переключатель режима** в чате — Ask / Agent / Plan / Debug
5. **Accept / Reject** — вы решаете, что попадает в код

Горячие клавиши (Mac / Win):
- Чат агента: `Cmd+I` / `Ctrl+I`
- Inline-правка: `Cmd+K` / `Ctrl+K`
- Command Palette: `Cmd+Shift+P` / `Ctrl+Shift+P`

---

# Демо: сценарий на 5 минут

**1. Ask** — «Объясни, что делает этот файл / модуль»  
→ показать: ответ без правок

**2. Agent** — маленькая задача: «Добавь кнопку / поправь текст / напиши функцию»  
→ показать: поиск файлов → diff → Accept

**3. Plan** — чуть шире: «Хочу добавить X, предложи план»  
→ показать: план → согласие → правки

**4. (опционально) Debug** — воспроизвести ошибку из консоли  
→ показать: гипотезы → фикс

*Совет спикеру: берите знакомый маленький репозиторий, без секретов.*

---

# Как писать промпты (главное)

Плохо: «почини авторизацию»  
Лучше: контекст + цель + ограничения + критерий готово

**Шаблон**

```
Контекст: ... (где в проекте, какой стек)
Задача: ...
Ограничения: не трогай ..., используй наш стиль ...
Готово когда: ... (что проверить)
```

**Ещё сильнее**
- Укажите файлы: `@файл` / `@папка`
- Приложите ошибку целиком (stack trace)
- Для большой задачи сначала: «только план, код не пиши»

---

# Примеры: слабо → сильно

**Слабо:** «Сделай форму логина»

**Сильнее:**  
«В `@src/pages/Login.tsx` добавь валидацию email и пароля (мин. 8 символов).  
Стили как у соседних форм. Тесты не пиши. Готово: форма не сабмитится при ошибке.»

**Слабо:** «Почему падает?»

**Сильнее:**  
«После деплоя 500 на `/api/orders`. Лог: *[вставить]*.  
Режим Debug: найди причину, предложи минимальный фикс.»

---

# Ещё примеры: слабо → сильно

| Слабо | Сильно |
|---|---|
| «Сделай сайт быстрее» | «Страница `/catalog` грузится 4с. Найди узкое место и предложи 2–3 варианта ускорения» |
| «Добавь тесты» | «Напиши unit-тесты для `@utils/price.ts`, покрой крайние случаи: 0, отрицательные, null» |
| «Переведи на английский» | «Переведи все тексты интерфейса в `@src/locales/ru` на английский, создай `en`, ключи не меняй» |
| «Сделай безопасно» | «Проверь `@api/auth.ts` на утечки токенов и SQL-инъекции, предложи фиксы» |

**Вывод:** конкретика + файл + критерий «готово» = хороший результат.

# Ключевая мысль: как ИИ вас «слышит»

⚠️ Если вы напишете **уверенную команду без сомнений** — ИИ сделает **ровно это**, топорно, и **не будет искать решение сам**.

**Пример (вы не уверены, но пишете как приказ):**
> «Добавь кэш в Redis для списка заказов»

→ Он молча добавит Redis, даже если это лишнее.

**Лучше — оставить ему пространство подумать:**
> «Список заказов грузится медленно. Думаю, поможет кэш, но **не уверен**. Предложи варианты, выбери лучший и объясни почему.»

→ Теперь он рассуждает, сравнивает, а не слепо выполняет.

---

# Почему говорить «задавай вопросы»

ИИ **боится переспрашивать** и часто **додумывает** за вас — а домыслы бывают неверные.

Добавляйте в конце запроса:
> «Если чего-то не хватает — **задай уточняющие вопросы** перед тем, как писать код.»

**Что это даёт:**
- Он вытащит из вас недостающий контекст
- Меньше переделок «он понял не так»
- Вы сами лучше формулируете задачу, отвечая на вопросы

*Особенно важно для больших и размытых задач.*

---

# Почему не жалеть времени на план

Большая задача без плана = агент уходит делать **не то**, вы теряете время на откат.

**5 минут на план экономят полчаса переделок.**

Как правильно:
1. Режим **Plan** → «предложи план, **код пока не пиши**»
2. Читаете план как ТЗ: те ли шаги? ничего лишнего?
3. Правите: «шаг 3 не нужен», «добавь обработку ошибок»
4. Только после согласия → «ок, реализуй»

План — это где **дёшево** передумать. В коде передумывать дорого.

---

# Как разговаривать с ИИ — итог

- Не уверены в решении? **Так и скажите** — пусть предложит варианты
- Просите **«задавай вопросы»**, если задача размытая
- Большое → сначала **план**, потом код
- Давайте **контекст и файлы** (`@`), а не догадки
- Формулируйте **критерий готово**
- Итеративно: маленькими шагами лучше, чем один огромный запрос

**Правило:** относитесь к ИИ как к толковому джуну — чем яснее ТЗ, тем лучше результат.

---

# Полезные возможности в IDE

- **`@`** — подтянуть файл, папку, доки, git
- **Rules** (правила проекта) — «всегда так пиши / наш стек»
- **Diff review** — смотрите каждое изменение
- **Терминал** — агент может гонять команды (вы видите вывод)
- **Несколько чатов** — разные задачи = разные треды

Не обязательно всё сразу: начните с Ask + Agent и `@файлов`.

---

# Токены — что это

**Токен** — кусочек текста (примерно ¾ слова). ИИ читает и пишет **токенами**.

- Ваш запрос, файлы, ответы — всё это токены
- У модели есть **контекстное окно** — лимит, сколько влезает за раз
- Больше токенов = медленнее, дороже и выше шанс, что ИИ «забудет» начало

Аналогия: контекст — это **рабочий стол**. Маленький стол завалишь — начнёшь терять бумаги.

---

# Как экономить токены (и деньги)

- **Указывайте файлы через `@`** вместо «посмотри весь проект»
- **Новый чат под новую задачу** — старый контекст не тянется мёртвым грузом
- Не вставляйте **гигантские логи** целиком — только нужный кусок
- Формулируйте **чётко с первого раза** — меньше кругов переписки
- Большую задачу — **на подзадачи**, а не один мега-промпт
- Не держите **сотни файлов** открытыми в контексте без нужды

**Итог:** меньше «воды» → быстрее, дешевле, точнее ответы.

---

# Репозитории: где живёт код

**Репозиторий** — папка проекта + вся история изменений (через **Git**).

Хранят их в облаке, чтобы работать командой и не терять код:

| Платформа | Особенность |
|---|---|
| **GitHub** | Самая популярная, много готовых решений |
| **GitLab** | Часто в компаниях, свой CI/CD |
| **Bitbucket** | Экосистема Atlassian (Jira) |
| **Свой сервер** | Self-hosted, когда нужен полный контроль |

Cursor умеет работать с любым: коммиты, ветки, PR — прямо из редактора.

# Готовые решения: не изобретайте велосипед

Почти всё уже придумано — и Cursor поможет это подключить.

| Задача | Готовые решения |
|---|---|
| **Безопасность** | Аутентификация (Clerk, Auth0), сканеры уязвимостей |
| **Тесты** | Jest, Pytest, Playwright — фреймворки для проверки |
| **Деплой** | Vercel, Docker, CI/CD — выкладка «в один клик» |
| **База данных** | Postgres, Redis и готовые интеграции |

🎙️ «Не знаете, что выбрать? Спросите Cursor — он подскажет и настроит.»

---

# Главная установка: просто спросите

**Любой** вопрос по вашему проекту можно задать Cursor.

- «Как это работает?» → объяснит
- «Смени язык интерфейса на английский» → сделает
- «Это безопасно?» → проверит
- «Как задеплоить?» → распишет шаги
- «Есть ли готовое решение?» → предложит

Не сидите в тупике 2 часа. **Сначала спросите** — часто ответ за 30 секунд.

# Cursor vs Claude Code (коротко)

| | **Cursor** | **Claude Code** |
|---|---|---|
| Формат | IDE + агент в редакторе | CLI / терминальный агент |
| Удобно когда | Смотрите код, diff, UI рядом | Живёте в терминале |
| Сильная сторона | Visual review, привычный редактор | Сильный «агентный» цикл в CLI |
| Слабее | Иногда «шумит» UI; нужен review | Меньше «IDE-ощущения» из коробки |

У нас **Cursor** — учимся выжимать его в ежедневной работе.  
Инструменты дополняют друг друга; выбор — про привычный workflow.

---

# Типичные ошибки новичков

1. Слишком общая задача без контекста
2. Сразу Agent на большой рефакторинг без Plan
3. Accept всего diff’а не глядя
4. Секреты (`.env`, ключи) в чат / в коммит
5. Один гигантский чат на всё подряд — лучше новый тред на задачу

# Шпаргалка на день

| Хочу… | Делаю |
|---|---|
| Понять код | **Ask** + `@файл` |
| Маленькую правку | **Agent** + чёткий критерий |
| Большую фичу | **Plan** → потом Agent |
| Разобрать баг | **Debug** + лог/шаги |
| Контроль | Читаю diff, Accept выборочно |

**Золотое правило:** вы — владелец кода; агент — ускоритель.

---

# Итог

1. Агент ≠ чат: он работает **в вашем проекте**
2. Режим = стиль работы (Ask / Agent / Plan / Debug)
3. Хороший промпт: контекст → задача → ограничения → «готово когда»
4. Всегда review diff
5. Начните с малого — одна понятная задача в день

Вопросы? Давайте попробуем задачу из вашего текущего проекта.
